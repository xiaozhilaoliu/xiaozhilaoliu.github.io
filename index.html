面向对象六大原则

发表于 2017-03-06 | 分类于 项目,设计模式

一、面向对象的六大原则

1、单一原则S

  • 英文名称：Single Responsibility Principle;缩写是SRP;
  • 定义：就一个类而言，应该仅有一个引起他变化的原因；

2、开闭原则O

  • 英文名称：Open Close Principle，缩写是OCP；
  • 定义：软件中的对象（类、模块、函数等）应该对于扩展开放，但是，对于修改是封闭的。
  • 现状：当软件需求发生变化时，我们应该尽量通过扩展的方式来实现变化，而不是通过修改已有的代码来实现。方然，在实际开发中，修改原有代码、扩展代码往往是同事存在的。

3、里氏替换原则L

  • 英文名称：Liskov Sybstitution Ptoncipal,缩写是LSP;
  • 定义
    • 第一种：如果对每一个类型为S的对象O1，都有类型T的对象O2，使得以T定义的所有程序P在所有的对象O1都替换成O2时，程序P的行为没有发生变化，那么类型S是类型T的子类型。
    • 第二种：所有引入基类的地方必须能够使用其子类对象；

4、依赖倒置原则D

  • 英文名称：Dependence Inversion Principle;缩写是DIP;
  • 定义：依赖倒置原则指代了一种特定的解耦形式，使得高层次的模块不依赖与低层次的模块的实现细节的目的，依赖模块被颠倒了。
  • 关键点：
    • 高层模块不应该以来低层次模块，两者都应该以来其抽象。
    • 抽象不应该依赖细节。（抽象值接口或抽象类）
    • 细节应该依赖抽象。
  • java中的表现

模块之间的以来通过抽象发生，实现类之间不直接发生关系，其依赖关系是通过接口或者抽象类产生的。

5、接口隔离原则I

  • 英文名称：InterfaceSegregation Principles
  • 定义：
    • 第一种：客户端不应该依赖他不需要的接口。
    • 第二种：类间的依赖关系应该建立在最小的接口上。
  • 目的
    • 臃肿的接口拆分成更小个更具体的接口，使系统解耦，容易重构、更改和重新部署。
    • 让以来的接口尽可能的小。

Bob大叔在21世纪早期，将单一原则、开闭原则、里氏替换、接口隔离以及以来倒置5个原则定义为SOLID原则，作为面向对象的5个基本原则

6、迪米特原则

  • 英文名称：Law of Demter
  • 定义：一个对象应该对其他对象有最少的了解。通俗的的讲，一个类应该对自己需要耦合或者调运的类知道最少，类的内部如何实现与调运着或者依赖着没有关系，调运者或者依赖着只需知道他需要的方法即可，其他的一概不管。类与类之间知道的越多，耦合就越大，当一个类发生改变时，影响也就越大。

Android Studio关联redmine

发表于 2017-02-19 | 分类于 项目
  • 为什么要关联
    • 可否在我们的开发工具中查看以及处理分配到自己的issue，而不需要打开redmine工具？
    • 对于重大的issue，需要创建分之修复，是否有办法将issue和分之对应，便于我们管理？

  • 怎么做

    首先，关于上面的疑问，答案是肯定的，有方法，而且也特别简单，那就
是Android Studio自带的工具。
    • 步骤一：打开server配置，打开Android Studio，Tools->Tasks&Contexts->Configure Servers；
    • 步骤二：添加redmine服务点击配置面板右上角的+号，选择Redmine；
    • 步骤三，配置账号，这里很简单，就不详细说了
    • 配置完成，点击test，测试一下是否成功。
    • 查看，分配到自己的issue，Tools->Tasks&Contexts->Open Task;
    • 关联issue和分支，点击想关联的issue，Create branch为你想要命名的分之名字。
    • 切换到创建的任务分之，Tools->Tasks&Contexts->Switch task;这里显示关联的分之，可以快速切换；
    • 修复完成之后，可以关闭任务！Tools->Tasks&Contexts->Close Active task，他就会将代码提交，并且合并到主分支（主分支就是你创建bug分之的源分之）
  • 带来的好处
    • 方便查看分配到自己的issue
    • 修复bug，关联分支，不影响现有代码和分支
    • 修复完成之后，关闭任务，自动提交和合并代码！

    2. 敏捷开发，既然是开发，肯定离不开一个概念，开发，开发就是为了发布，那么，基于现在的打包流程，android studio，打一个包，需要五分钟左右，电脑差点，十分钟也有可能，然后在上传到fir，这本是是一个相对简单的流程；但是就是他，会引起很多问题，譬如：你可能会打断一个正在开发中的程序员、每个程序员使用的环境不一样、每次打包可能对应的代码不一致、每次都是手动打包上传（low不low，是不是该把程序员的精力放在更有意义的事上面）、版本号怎么维护、每次打包依靠手动修改versioncode（如何保证自增呢），基于种种问题，我根据以前的一些实践，搭建了一套粗陋的持续构建框架。
    3. 我的持续构建框架，整体是基于go cd的，在搭建过程中，是遇到很多坑的，本人并非运维出身，对于linux也是熟悉，根本达不到熟练的水平；其中就遇到过，go cd无法clone代码，明明加了权限也不行，go cd只是一个ci架构，具体的打包，还是需要自己来写脚本，那么就查找各种资料，学习shell的相关知识，遇到最多的也是权限问题和环境问题；打包脚本搞定了，接下来就是，我如何保证versioncode自增，verisonname可以方便自定义呢？我的解决方案是，verisoncode持久化在一个文件中，每次打包自增然后保存，versionname通过go cd以参数的方式传给打包脚本；打包脚本，再把versioncode和versionname，以参数方式传给gradle，gradle中，会读取这两个参数，如果没有参数，就会使用默认；这样好处就是，既可以使用go cd打包，也不影响我们编译运行，一举两得；环境中，遇到最麻烦的事，就是在centos中，使用android sdk打包，他会有缺少glib-2.14库缺少的问题，这个也用了很长时间解决，就赢个这个，弄坏了好几台服务器，这里特别感谢我们的运维兄弟；
    4. 持续化环境搭建的好处
      • 同样的环境，同样的代码
      • 操作简单，主要有权限，都可以执行
      • 打包发布一步到位，这个才对得起敏捷开发这个词。
      • 对于多环境的打包，区分配置，不容易混淆。
      • 个人观点就是，能用工具的，就不要人力去做了，太累了，而且人犯错率比较高。总之，就是提高效率
    5. 以后可优化的点
      • 速度采用feeline或者okbuck，提升打包速度
      • 打包自动化，增加预警和提示功能，（比如规定每天凌晨打包，每次上午八点，将新版本地址发到测试人员和灰度测试人员手中。）
      • 代码质量管理，对于提交的代码，自动进行审核，生成报告，提交到审核人员手中。
      • 自动化测试，对于每天晚上的自动化打包，执行自动化测试，生成报告，发给相关人员
      • 单元测试，代码中大量使用单元测试，每一次修改功能，都会自动执行单元测试，如果不通过，发出预警。单元测试的好处

重构代码读书笔记

发表于 2016-11-16 | 分类于 读书笔记

重构的第一步

重构中，有几个很重要的原则和注意点。
  • 如果你发现自己需要为程序添加一个特性，而代码结构使你无法很方便地达成目的，那就先重构那个程序，使特性的添加比较容易进行，然后再添加特性。
  • 重构前，先检查是否有一套可靠的测试机制。这些测试必须有自我检验能力。
  • 重构技术就是以微小的步伐修改程序。如果你犯下错误，很容易便可发现它。
  • 任何一个傻瓜都能写出技术及可以理解的代码。唯有写出人类容易理解的代码，才是优秀的程序员。

重构原则

  • 何谓重构

    有一个很重要的原则，重构不会改变软件可观察的行为--重构之后软件功
能一如以往。任何最终用户或者其他程序员，都不知道已经有东西发生了
变化。
    • 名词：对软件内部结构的一种调整，目的是在不改变软件可观察行为的前提下，提高其可理解性，降低其修改成本。
    • 动词：使用一系列重构手法，在不改变软件可观察行为的前提下，调整其结构。
  • 为何重构
    • 重构改进软件设计
    • 重构使软件更容易理解
    • 重构帮助找到bug
    • 重构提高编程速度

  • 何时重构

    重构是不需要专门安排时间来进行重构，答案是否定的；有一个注明
的`三次法则`，事不过三，三则重构；也就是，第一次做某件事情只管去
做；第二次做类似的事情会产生反感；第三次再做类似的事情，就应该重
构。
    • 添加新功能时重构
    • 修改错误时重构
    • 复审代码时重构

优雅的处理错误

发表于 2016-10-25 | 分类于 代码优化

why

  • 许多程序，我们会设计的错误处理和异常处理，简单的登录，各种异常信
    息，面貌错误、手机号检测等等；但是如果这些处理不当，它就会搞乱你
    的代码逻辑，会导致代码的可读性变得很差。

how

  • 使用异常而非返回码

    简而言之，如果有诸多错误信息，你通过状态码来处理和识别，会导致时
    间异常，你根本无法区分这些错误是什么，所以时候用一个表意更清楚的
    异常类，显得简单可靠

  • 给出异常发生的环境说明

    每个抛出的异常，都应该提供左后的环境说明，以便于判断错误的来源和
    出处

  • 调用者需要定义异常类

  • 别返回null值

    返回null这样通常都会引起空指针异常，如果上层代码没有捕获这个异
    常，就会引起奔溃。

  • 别传递null

    传递null，同样也会在不严谨的代码中导致奔溃。

二、敏捷开发的一些感悟

      实际上更确切的说法是敏捷方法强调适应性而非预见性’，我非常同意这句话，各别公司对于敏捷的认识比较肤浅，
      当然，我本人也理解的也是很一般；
    • 首先说适应性，也就是说，敏捷可以很快的调整计划，快速制定新的发布计划
      和发布方案，也就是从需求提出->产品确认->开发确认可行性，然后结合研发人员的时间评估，与项目经理的意见
      相结合，制定新的发布方案和发布计划，也就是说，拥有变化，能很好地适应各种需求的变更，也符合互联网
      周期短的特性，想一想，当你产品做到一半，发现原先的产品需要有需求的变更，如果利用敏捷开发，就能够
      快速的调整改变；
    • 其次是非预见性，也就是，一个项目的进展，毕竟存在很多未知的因素，虽然前期的准备
      工作，能够降低未知因素，但是，绝对不会说是完全消除；很简单，比如，有一次，我们做好了
      一次准备工作；发布计划、发布方案、详细技术设计、环境部署，全部很ok，但是最终，只是
      因为需求方的一个需求，线上的版本，可以动态切换环境，就由于这个简单的需求，需要
      我们重新设计，开发，测试，以及部署，这个过程绝对不会小于2个工作天，而且牵连也很大，
      而对于敏捷，我们这个敏捷团队，就针对这个变化，快速调整，快速制定，快速开发，快速上线，
      而不是和以往的流程，比较依赖固定的计划和文档，对于一些变更的内容，修改起来，牵扯的关系很复杂，
      很难刀刀这一目的。
